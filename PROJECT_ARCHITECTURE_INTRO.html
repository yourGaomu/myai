    Spring Boot Mental Model

    用你熟悉的分层方式阅读 Go 项目

    接口、输入对象、结果对象和实现类已经物理隔离。不同之处只是 Go 更强调小接口、显式装配和单向依赖。

    01cmd / remote接收 CLI、HTTP、WebSocket 请求，转换协议参数Controller
    02service.ChatService给所有入口提供稳定、统一的业务门面Facade
    03applicationapi 定义能力，service 实现业务，command/result 承载数据Service + DTO
    04port描述应用层需要的数据库、模型、缓存等出站能力Repository Interface
    05adapter实现 Mongo、Redis、LLM、工具、Hook 和外部服务RepositoryImpl

    Stable Prefix Strategy

    Plan 模式不会打乱固定缓存前缀

    模式切换只更新 Session 状态。构建本轮请求时，Plan 指令临时插入到最新用户消息之前，不会写回历史，也不会替换固定系统提示词。

    固定 system 和历史消息保持稳定；变化只发生在请求尾部，因此更利于 prompt cache 命中。
    SYSTEM固定的 MyAI 行为、工具与安全规则
    HISTORY稳定的历史 user / assistant / tool 消息
    RUNTIME本轮 Plan 模式规则与匹配到的 Skill 指令
    USER本轮最新输入：请重构会话模块

    Three Rules

    快速记住整个项目

    看代码时先判断它属于入口、业务用例还是技术实现。只要这三个边界不混，项目就不会重新变乱。

    01 / EXECUTION

    Agent 才是真正执行端

    手机负责 UI，Relay 负责转发，模型、工具和文件操作都发生在 PC Agent。

    02 / BUSINESS

    业务流程放 Application

    ChatService 只是 Facade；真正的生成、Plan、Session 等流程由应用服务负责。

    03 / WIRING

    实现关系去 Composition 找

    当你不知道接口注入了谁，查看 composition/chat/configuration.go，而不是猜测运行时扫描。
